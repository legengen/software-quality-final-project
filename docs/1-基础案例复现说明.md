# 基础案例复现说明

本文档说明本项目如何复现 AGENTLESS 使用 LLM 修复 SWE-bench Lite 基础缺陷案例。

## 1. 目标

基础复现阶段的目标是验证 AGENTLESS 在本地环境中能否完成端到端自动程序修复流程：

```text
Issue / problem_statement
  -> file-level localization
  -> related-level localization
  -> fine-grain localization
  -> repair 生成补丁
  -> SWE-bench Docker evaluation
```

本项目没有重新实现 AGENTLESS 算法，而是在官方流程基础上接入 DeepSeek V4 Pro High，并选取 SWE-bench Lite 中 5 个 Astropy 案例进行复现。

## 2. 实验环境

| 项目 | 配置 |
| --- | --- |
| 系统环境 | WSL Linux |
| 项目目录 | `/home/legengen/software-quality-final-project` |
| AGENTLESS 目录 | `Agentless/` |
| 评测框架 | SWE-bench harness + Docker |
| 模型 | DeepSeek V4 Pro High |
| backend | `deepseek` |
| 数据集 | SWE-bench Lite 子集，本地 JSON 形式 |
| 代理 | WSL 通过 Windows 主机代理出口，端口 `40558` |

为保证本地可运行，本项目做了以下工程适配：

- 增加 DeepSeek backend 与 `deepseek-v4-pro` 模型选项；
- 为 repair 增加 `--max_tokens` 参数，避免输出预算不足导致空补丁；
- 支持从本地 JSON 文件加载 dataset；
- 为 Docker build 和 SWE-bench evaluation 透传代理环境变量；
- 增强补丁后处理，使其能解析部分 DeepSeek 生成的 `edit_file(...)` 格式。

## 3. 基础复现流程

每个基础案例按照 AGENTLESS 官方思路执行：

1. 文件级定位：根据 issue 描述和仓库结构定位可能需要修改的文件。
2. 相关元素定位：在目标文件中定位相关类、函数或方法。
3. 细粒度定位：进一步定位具体行号或代码块。
4. 补丁生成：调用 DeepSeek V4 Pro High，根据定位上下文生成修复补丁。
5. 补丁后处理：将模型输出转成 SWE-bench 可读取的 unified diff JSONL。
6. 官方评测：使用 SWE-bench Docker harness 运行 FAIL_TO_PASS 和 PASS_TO_PASS 测试。

## 5. 测试反馈迭代

其中 3 个案例通过初始 repair 直接通过。`astropy__astropy-14182` 和 `astropy__astropy-14365` 初始补丁未完全通过官方测试，后续采用测试反馈驱动的迭代修复：

- `astropy__astropy-14182`：根据测试失败信息，引导模型使用 `self.header.header_rows`，并修正 `self.data.start_line` 的计算方式，最终通过。
- `astropy__astropy-14365`：根据 `ValueError: could not convert string to float: 'no'` 的失败信息，引导模型同时处理小写 `no`，最终通过。

该过程没有直接使用 gold patch，而是根据官方评测失败日志提炼反馈，再让 DeepSeek 重新生成候选补丁。

## 6. 结论

基础复现证明 AGENTLESS 的定位、补丁生成和官方测试评测流程可以在本地 WSL + Docker 环境中跑通。DeepSeek V4 Pro High 能够替代 OpenAI-compatible backend 完成 5 个 Astropy 基础案例修复，最终结果为 `5 resolved / 0 unresolved`。

同时，实验也说明单次 repair 可能生成语义不完整的补丁。对失败案例加入测试反馈后，模型可以根据具体错误信息进一步修正补丁，提高最终通过率。
