# 并发代码修复与创新点

本文档说明本项目如何在 AGENTLESS 基础复现之上加入自定义并发缺陷修复流程，以及围绕并发代码修复增加的轻量创新点。

## 1. 自定义并发案例

自定义项目位于：

```text
benchmarks/concurrent_counter/
```

核心缺陷位于：

```text
benchmarks/concurrent_counter/concurrent_counter/counter.py
```

`ConcurrentCounter.increment()` 存在典型读-改-写竞态：

```python
old_value = self._value
time.sleep(0.00001)
self._value = old_value + amount
```

多个线程可能同时读取相同旧值，再分别写回 `old_value + amount`，导致最终计数小于期望值。

## 2. SWE-bench 风格包装

为使自定义并发缺陷接入 AGENTLESS 流程，项目将其包装成 SWE-bench 风格 case。生成脚本为：

```bash
python3 scripts/generate_custom_concurrency_case.py
```

生成的关键资源：

```text
Agentless/resources/custom_concurrency/concurrency_cases.json
Agentless/resources/custom_concurrency/project_structures/local_concurrency__counter-0001.json
Agentless/resources/custom_concurrency/gold_patch.jsonl
```

case 中包含 `problem_statement`、`test_patch`、`FAIL_TO_PASS`、`PASS_TO_PASS` 和项目结构快照。

## 3. 从定位到修复的流程

推荐使用一键脚本演示完整流程：

```bash
cd /home/legengen/software-quality-final-project
export OPENAI_API_KEY=你的DeepSeek_API_Key
REPEAT=10 scripts/run_concurrency_repair_demo.sh
```

脚本执行流程：

```text
生成自定义并发 case
  -> file-level localization
  -> related-level localization
  -> fine-grain localization
  -> 定位结果规范化
  -> DeepSeek repair
  -> evaluator 多轮 pytest 评测
```

默认输出目录会带时间戳，例如：

```text
Agentless/results/custom_concurrency/counter_live_demo_20260627-142500/
```

关键输出文件包括 `loc_outputs.jsonl`、`loc_outputs.normalized.jsonl`、`repair/output_0_processed.jsonl` 和 `evaluation_report.json`。

## 4. 本地 evaluator

由于自定义本地项目不属于 SWE-bench 官方 benchmark，不能直接复用官方 Docker harness。因此项目实现了本地 evaluator：

```text
scripts/evaluate_custom_concurrency.py
```

评测流程：

```text
复制 benchmarks/concurrent_counter
  -> git init
  -> 应用 test_patch
  -> 应用 model_patch
  -> 重复运行 pytest
  -> 输出评测报告
```

## 5. 创新点一：并发稳定性评测

原始测试流程通常只运行一次 pytest，并给出 `resolved / unresolved`。对于并发缺陷，单次运行结果受线程调度影响，不能充分说明补丁是否稳定。

evaluator 新增：

```bash
--repeat N
```

对同一补丁重复运行测试，并输出：

| 字段 | 含义 |
| --- | --- |
| `repeat` | 重复运行次数 |
| `passed_runs` | 通过轮数 |
| `failed_runs` | 失败轮数 |
| `pass_rate` | 通过率 |
| `stable_resolved` | 是否每轮都通过 |

当前结果：

| patch | repeat | passed_runs | failed_runs | pass_rate | stable_resolved |
| --- | --- | --- | --- | --- | --- |
| no-op | 10 | 0 | 10 | 0.0 | false |
| gold | 10 | 10 | 0 | 1.0 | true |

该结果说明并发测试能稳定暴露原 bug，也能验证正确加锁补丁。

## 6. 创新点二：并发语义增强 Prompt

原 AGENTLESS repair prompt 是通用代码修复提示，没有专门强调共享状态、读-改-写、临界区和锁覆盖范围。

在 `Agentless/agentless/repair/repair.py` 中增加：

```bash
--concurrency_hint
```

启用后，repair 阶段会在 issue 描述后追加并发修复提示，提醒模型关注共享可变状态、读-改-写序列、临界区、锁覆盖范围、顺序行为保持，以及不要通过删除或削弱测试来解决问题。

该参数只用于自定义并发流程，不影响 5 个基础 SWE-bench 案例的复现结果。

## 7. 创新点三：补丁并发风险检查

仅靠 pytest 结果无法解释补丁是否符合并发修复模式。例如，补丁可能只改测试、加入 sleep，或者没有保护所有共享状态修改路径。

`scripts/evaluate_custom_concurrency.py` 中新增 `patch_risk` 静态分析，检查：

| 字段 | 含义 |
| --- | --- |
| `changed_source_files` | 是否修改源文件 |
| `changed_test_files` | 是否修改测试文件 |
| `uses_lock` | 是否使用 `Lock` / `RLock` |
| `uses_context_manager` | 是否使用 `with self._lock` |
| `guards_increment` | 是否保护 `increment()` |
| `guards_reset` | 是否保护 `reset()` |
| `patches_tests_only` | 是否只修改测试 |
| `adds_sleep` | 是否新增可疑 sleep |
| `risk_level` | 风险等级：`low` / `medium` / `high` |
| `risk_reasons` | 风险原因列表 |

当前对比：

| patch | uses_lock | guards_increment | guards_reset | patches_tests_only | risk_level |
| --- | --- | --- | --- | --- | --- |
| no-op | false | false | false | false | high |
| gold | true | true | true | false | low |

## 8. 与官方基础案例的差异

| 维度 | 基础 SWE-bench 案例 | 自定义并发案例 |
| --- | --- | --- |
| case 来源 | SWE-bench Lite Astropy | 本地构造 |
| 被测项目 | Astropy | `concurrent_counter` |
| 测试来源 | 官方 FAIL_TO_PASS / PASS_TO_PASS | 自定义 pytest 测试 |
| 评测方式 | SWE-bench Docker harness | 本地 evaluator |
| repair prompt | 通用修复提示 | 增加 `--concurrency_hint` |
| 结果指标 | resolved / unresolved | resolved、pass_rate、stable_resolved、patch_risk |

## 9. 结论

并发扩展保留了 AGENTLESS 的核心思路：先定位，再基于上下文生成补丁，最后用测试验证。针对并发缺陷的特点，项目增加了多轮稳定性评测、并发语义提示和补丁风险检查，使流程更适合展示线程竞态类 bug 的自动修复效果。
